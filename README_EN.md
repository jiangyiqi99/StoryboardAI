# Storyboard AI Editor

[Chinese](README.md) | [English]

An AI-native local non-linear video editor scaffold. At this stage, the project only establishes the Electron + React + TypeScript + Vite structure, core module placeholders, type definitions, IPC call flow, and project-file documentation. It does not implement production UI details, real model calls, complex FFmpeg commands, a complete editing algorithm, or cloud services.

## Design Goals

- Desktop-first: a cross-platform Electron app, not a traditional SaaS admin panel.
- Local-project-first: `.aivproj/` folders and `project.json` are the center of project state.
- Timeline-first: tracks, clips, assets, selections, and edit history are the core model.
- Localized media processing: preview, frame, and render engines are separated. The first stage uses placeholders/mocks and FFmpeg/FFprobe call slots.
- Native Media Runtime: adds a Go/libav sidecar contract as a parallel path for future native decoding, preview, and export. It has not replaced the existing Preview or FFmpeg paths.
- AI-native: storyboard-to-video generation and selected-range replacement are first-class workflows, not add-on plugins.
- Non-destructive editing: timeline clips reference media assets and do not rewrite original video files.

## Development and Packaging

```bash
npm run dev
npm run build
./build --macarm64
./build --macx64
./build --win64
./build --linux
```

`npm run build` only writes the Electron/Vite runtime output to `out/`. `npm run dist:*` first builds the libav sidecar for the current architecture, collects its FFmpeg dynamic libraries and dependencies into `native/libav/runtime/`, and then uses electron-builder to create installers in `release/`. The runtime directory is bundled into the application resources, so target machines do not need a separate FFmpeg installation.

Cross-platform releases are built in GitHub Actions on macOS arm64, macOS x64, Windows x64, and Linux x64 runners. Each push creates four groups of Actions artifacts. App icons can be added later at the electron-builder-supported paths `build/icon.icns`, `build/icon.ico`, and `build/icon.png`.

## Core Layers

### 1. Renderer UI

Location: `src/renderer`

The renderer is only responsible for the desktop editor interface and user-action entry points:

- `timeline`: UI placeholder for tracks, clips, playhead, and selection.
- `preview`: preview display surface. In the first stage it can connect to an `HTMLVideoElement` or mock preview engine.
- `media-bin`: asset list and asset-registration entry point.
- `inspector`: property panel for the selected clip, range, or asset.
- `storyboard`: entry point for storyboard scripts and segment management.
- `replace-selection`: entry point for replacing a timeline range.
- `export`: rendering/export entry point.

The renderer does not directly read or write arbitrary files, call FFmpeg, or hold third-party model API keys. All local permissions go through the typed IPC API exposed by preload as `window.aiv`.

### 2. Electron Main Process

Location: `src/main`

The main process is responsible for:

- Registering IPC: `project:create/open/save`, `media:probe/extractFrame/renderTimeline`, and `ai:generateStoryboard/replaceRange/getJobStatus`.
- Safe file access: future expansion through file pickers, path allowlists, and `.aivproj` root validation.
- Calling the Local Project System.
- Calling the Local Media Engine.
- Calling the AI Orchestrator.

Call direction:

```text
Renderer UI
  -> preload window.aiv
  -> Electron IPC
  -> Main Process handlers
  -> Project System / Media Engine / AI Orchestrator
  -> project.json + local media outputs
```

### 3. Local Project System

Location: `src/project-system`

`.aivproj/` is the project boundary:

```text
MyFilm.aivproj/
├── project.json
├── assets/
├── frames/
├── cache/
├── proxies/
├── thumbnails/
├── renders/
└── ai/
```

`project.json` stores:

- Project metadata: `id`, `name`, `schemaVersion`, created/updated timestamps.
- Project settings: resolution, fps, audio sample rate, color space, and preview quality.
- Asset references: `assets[]`, with project-relative paths and probed metadata.
- Timeline: `tracks[]`, `clips[]`, playhead, and selection.
- Storyboard: `storyboardSegments[]`.
- AI jobs: `aiGenerationJobs[]`.
- Render cache: `renderCache[]`.
- Edit history: `editHistory.past/future`.

This layer handles path management, asset registration, cache management, and project-file reads/writes. Project files are read strictly according to the current schema. Release builds do not keep migration paths for old schemas.

### 4. Local Media Engine

Location: `src/media-engine`

The media engine is split into three roles:

- Preview Engine: `seek`, `play`, `pause`, `scrub`, and low-resolution preview. The first stage uses a mock/`HTMLVideoElement` direction. It may later be replaced by mpv/libmpv, a Go video-processor binary, C++/libav, or MLT.
- Frame Engine: `probe`, `seek`, `extractFrame`, `extractFirstLastFrames`, `thumbnail`, and `proxy`. The first stage reserves Node `child_process` calls to FFmpeg/FFprobe.
- Render Engine: `trim`, `normalize`, `concat`, `renderSelection`, `renderTimeline`, `renderReplacementRange`, and `exportTimeline`. The first stage only keeps interfaces and call slots.

Preview is not Render: preview may use proxies, lower resolution, or approximate playback. Final export goes through the Render Engine and produces stable output from timeline data and project settings.

### Native Media Runtime (Experimental)

Locations: `src/main/native-media`, `src/shared/types/native-media.ts`, and `native/libav`.

The existing `media:*` IPC, `MockPreviewEngine`, `FfmpegFrameEngine`, and `FfmpegRenderEngine` are unchanged. The new channel is exposed as `window.aiv.nativeMedia` and provides `openAsset`, `probe`, `decodeFrame`, playback session controls, `renderFrame`, and `encodeTimeline`. The main process lazily starts the Go sidecar. Control messages use stdio JSON-RPC. Large frame/audio interfaces are defined as shared-memory leases to avoid copying video frames through JSON. The current native decoder first uses inline data under the same contract to validate libav decoding; the platform shared-memory allocator is not enabled yet.

Frame conventions support `rgba`, `bgra`, and `yuv420p`, including width/height/stride, planes, PTS/timebase, duration, color space, and alpha/opacity. Audio conventions support `f32le` and `s16le`, including sample rate, channel count, sample frames, PTS/timebase, and the same transport.

Building the native binary requires Go and system libav development libraries:

```bash
npm run native:libav
```

See [native/libav/README.md](native/libav/README.md). If `nativeMedia` is called before the binary is built, it reports a clear runtime-unavailable error. The app's existing media capabilities are not affected.

When publishing GPL builds, you must also provide the exact corresponding FFmpeg source, configuration parameters, and this project's source code for the packaged FFmpeg dynamic libraries. GitHub Releases should attach those corresponding sources or provide clear, free source links from the same download location.

### 5. AI Orchestrator

Location: `src/ai-orchestrator`

The AI Orchestrator owns two core workflows, but does not directly depend on any specific third-party model. It only enters the AI API Routing Layer through `ApiRouter.generateVideo()`, `ApiRouter.getJobStatus()`, and `ApiRouter.cancelJob()`.

Storyboard script to video:

```text
script
  -> split into StoryboardSegment[]
  -> first segment: ApiRouter.generateVideo(text-to-video)
  -> extract previous tail frame
  -> following segments: ApiRouter.generateVideo(first-frame-to-video)
  -> register generated Asset
  -> insert non-destructive Clips into Timeline
  -> save project.json
```

Selected-range replacement:

```text
TimelineRange selection
  -> duration = range.end - range.start
  -> FrameEngine extracts first/last frames
  -> ApiRouter.generateVideo(replace-range)
  -> RenderEngine.normalize generated output
  -> register generated Asset
  -> create REPLACE_RANGE command
  -> split original clips into before / replacement / after
  -> save project.json
```

The replacement result is always a new generated asset plus updated timeline references. It never rewrites the original asset file.

### 6. AI API Routing Layer

Locations:

- `src/shared/ai-routing`: unified type contract that works across processes and can later move to the cloud.
- `src/main/ai-router`: router implementation scaffold in the Electron main process.
- `src/main/providers`: third-party model provider adapter placeholders.
- `src/main/model-proxy`: placeholder client for a future Thin Model Proxy.

Layer boundaries:

- The app only calls unified interfaces: `ApiRouter.generateVideo()`, `ApiRouter.getJobStatus()`, and `ApiRouter.cancelJob()`.
- `ApiRouter` chooses a provider adapter from unified parameters, route rules, provider, model, capabilities, and mode.
- `ProviderAdapter` handles third-party API field mapping, authentication shape, polling, cancellation, and response conversion.

The unified request type is `GenerateVideoRequest`. Core fields include:

```ts
interface GenerateVideoRequest {
  providerId?: string;
  modelId?: string;
  mode:
    | "text-to-video"
    | "image-to-video"
    | "first-frame-to-video"
    | "first-last-frame-to-video"
    | "video-to-video"
    | "replace-range";
  prompt: string;
  negativePrompt?: string;
  durationSec?: number;
  width?: number;
  height?: number;
  fps?: number;
  aspectRatio?: string;
  seed?: number;
  stylePreset?: string;
  cameraMotion?: string;
  referenceImages?: ReferenceImageInput[];
  firstFrameAssetId?: string;
  lastFrameAssetId?: string;
  inputVideoAssetId?: string;
  maskAssetId?: string;
}
```

The unified response type is `GenerateVideoResponse`, storing `jobId`, `providerId`, `providerJobId`, `modelId`, `mode`, `status`, `outputUri`, `error`, and the matched `route`.

Route selection flow:

```text
GenerateVideoRequest
  -> GenerateVideoRequestValidator
  -> AssetReferenceResolver
  -> ProviderRouteMatcher
  -> ProviderAdapter.mapRequest
  -> ProviderAdapter.submitGeneration
  -> ProviderAdapter.mapResponse
  -> GenerateVideoResponse
```

Capabilities matching:

- `mode` must be included in `ProviderCapabilities.supportedModes`.
- If `modelId` is specified, the model's `supportedModes` must also match.
- `durationSec` must fall inside the provider/model-supported range.
- `aspectRatio`, `fps`, `width`, and `height` are filtered by both provider route rules and capabilities.
- `negativePrompt`, `seed`, `referenceImages`, `firstFrameAssetId`, `lastFrameAssetId`, `inputVideoAssetId`, and `maskAssetId` are checked against provider-declared support.
- An explicit `providerId` takes priority. Otherwise the router selects by `ProviderRouteRule.priority`, then falls back to capabilities.

Parameter-mapping scaffold:

- `durationSec` may map to Runway `duration`, Kling `duration_sec`, Luma `generation.duration_seconds`, or Pika `seconds`.
- `width` / `height` / `aspectRatio` may map to provider-specific `resolution`, `ratio`, or `aspect_ratio`.
- `firstFrameAssetId` / `lastFrameAssetId` are first resolved by `AssetReferenceResolver` into uploadable files, temporary project URIs, or Thin Model Proxy-accessible URLs, and then passed by adapters into fields such as `keyframes`, `image`, or `references`.
- `inputVideoAssetId` / `maskAssetId` are used for video-to-video and replace-range input video/mask mapping.

Error-handling strategy:

- Parameter errors return `VALIDATION_ERROR`.
- No available provider returns `NO_ROUTE`.
- Provider or Thin Model Proxy failures return `PROVIDER_ERROR` or `PROVIDER_UNAVAILABLE`.
- Cancellation failure returns `CANCEL_FAILED`.
- The router returns a unified `GenerateVideoResponse` and does not leak raw provider errors to the renderer. Raw responses are kept in `rawProviderResponse` as a debugging placeholder.

Reserved providers:

- `MockProviderAdapter`
- `RunwayProviderAdapter`
- `KlingProviderAdapter`
- `LumaProviderAdapter`
- `PikaProviderAdapter`

These do not call real third-party APIs. They only demonstrate field mapping and job-lifecycle placeholders.

## Adding a New Model

Steps for adding a new provider:

1. Add a `ProviderAdapter` under `src/main/providers`.
2. Declare `ProviderCapabilities`, including modes, models, duration, aspect ratio, fps, reference-frame, mask, polling, and cancellation support.
3. Implement `mapRequest()` to convert the unified `GenerateVideoRequest` into provider-specific fields.
4. Implement scaffolds for `submitGeneration()`, `getJobStatus()`, and `cancelJob()`. Real secrets should preferably go through `src/main/model-proxy` or a future cloud Thin Model Proxy.
5. Register the adapter in `createDefaultProviderAdapters()`.
6. Add route rules and priority to `DEFAULT_PROVIDER_ROUTE_RULES`.

The Thin Model Proxy only protects third-party API keys, submits jobs, checks status, and cancels jobs. It does not store timelines, projects, or a user business database. The project source of truth remains the local `.aivproj/project.json`.

## Media Kernel Replacement Path

The current Frame/Render Engine only reserves imperative Node `child_process` + FFmpeg/FFprobe interfaces. Future replacement options:

- mpv/libmpv: replace the Preview Engine for more reliable seeking, playback, and scrubbing.
- Go video-processor binary: call a local binary from the Frame/Render Engine through a stable CLI or RPC.
- C++/libav: move probe, frame extraction, normalization, and render pipelines into a native addon or separate process.
- MLT: replace the Render Engine with a mature NLE graph/timeline/render pipeline.

As long as `MediaEngineFacade`, `FrameEngine`, and `RenderEngine` interfaces stay unchanged, the renderer and AI Orchestrator do not need to know the underlying implementation.

## Current Local Editor Features

The first renderer version includes basic editing interactions:

- Video, audio, and image import through the Media Bin import button or by dropping files into the Media Bin.
- Add assets to the timeline by dragging from the Media Bin into the Timeline, creating a local timeline clip.
- Timeline dragging: drag existing clips to change `timelineStart`.
- Selection syncing: clicking a Media Bin asset or Timeline clip updates the Inspector and Preview.
- Video preview playback: imported videos use `HTMLVideoElement` for preview playback as the first-stage Preview Engine implementation.

These features still run only in local frontend state. They are not yet written to `.aivproj/project.json`, and they are not connected to FFmpeg frame extraction, proxy files, real rendering, or an undo/redo command stack.
